# Codex 固定节点与飞书任务协同设计

## 1. 目标

将当前依赖本地常驻长连接进程的时间管理系统，调整为由 Codex 本地自动化在固定时间运行的个人执行负责人。

最终使用方式为：

1. N哥通过飞书任务查看和执行当天主任务与子任务。
2. N哥通过时间管理机器人一对一私聊提交临时想法、任务、会议、卡点和成果。
3. Codex 在固定时间读取新增私聊、飞书任务状态和项目知识库。
4. Codex 判断是否需要创建、更新、重排、延后、委派或拒绝任务。
5. Codex 只在需要说明、追问、催办、验收或复盘时通过机器人私聊回复。
6. 项目成果验收后更新知识库中的正式项目进度。

目标闭环为：

```text
项目目标
→ 固定节点生成与校准计划
→ 飞书主任务和子任务执行
→ 机器人一对一协调
→ 成果验收
→ 项目进度更新
→ 每日复盘
```

## 2. 设计原则

### 2.1 载体分工

| 载体 | 正式职责 | 不承担的职责 |
| --- | --- | --- |
| 飞书任务 | 主任务、子任务、负责人、时间、截止时间、完成标准、状态和成果引用 | 长篇计划说明、催办对话和项目知识沉淀 |
| 机器人一对一私聊 | 收集临时输入、解释计划、追问卡点、提醒进度、通知调整、验收反馈和复盘 | 充当重复的任务清单 |
| Codex 固定自动化 | 定时读取、判断、排程、同步任务、回复、验收和维护知识 | 常驻监听或代替用户执行高风险动作 |
| WPS/JXOS 知识库 | 项目目标、里程碑、交付项、正式进度、复盘和有效规则 | 运行锁、消息检查点和飞书密钥 |
| SQLite | 消息去重、节点运行、任务映射、排程版本、验收和失败恢复 | 取代项目 Markdown 成为项目事实源 |

### 2.2 核心边界

- 不再向群聊发送大段任务卡，群聊不作为主要入口。
- 不要求 Terminal 常驻，也不保持飞书 WebSocket 长连接。
- 正式运行只处理 N哥与时间管理机器人的一对一私聊。
- Mac 仍需开机、联网并能够运行 Codex 本地自动化。
- 正常推进时保持安静，不为了证明系统运行而反复发消息。
- 当前正在执行的任务不被普通新增事项打断。
- 对外发布、付款、合同、报价、退款、删除和正式承诺仍需人工确认。

## 3. 固定运行节点

系统每天在 Asia/Shanghai 时区运行七个节点：

| 时间 | 节点 | 核心职责 |
| --- | --- | --- |
| 08:00 | 今日任务分配 | 生成当天飞书主任务和子任务，发送今日总览 |
| 09:00 | 开工前校准 | 合并处理 08:00–09:00 新增私聊，必要时调整 10:00 后计划 |
| 12:00 | 上午进度核实 | 核实上午子任务和成果，安排 14:00 后执行 |
| 15:00 | 下午启动检查 | 判断 14:00 任务是否真正启动，处理卡点和拖延 |
| 18:00 | 白天成果验收 | 汇总白天成果，确定晚间唯一核心任务 |
| 21:00 | 晚间冲刺安排 | 推动 21:00–24:00 最后交付，不再塞入多个大任务 |
| 24:00 | 每日复盘 | 汇总最终状态，更新项目输入并生成次日排程依据 |

同一套节点执行器根据当前节点选择不同策略，不为每个时间复制一套独立业务逻辑。

## 4. 任务模型

### 4.1 主任务

每个主任务必须包含：

- 动作型标题。
- 所属项目。
- 项目、里程碑和交付项标识（项目任务必填）。
- 优先级。
- 建议开始时间和截止时间。
- 预计总时长，原则上不超过 120 分钟。
- 任务目的。
- 完成标准。
- 成果提交要求。
- 是否需要验收。

每天最多安排 5 个关键主任务。任务池充足时，个人 IP 至少 2 项、极享 OS 至少 2 项；两个项目各保留至少 120 分钟。总排程不超过可用时间的 70%。12:00–14:00 不安排执行任务。个人 IP 默认优先，只有影响极享 OS 正常使用的 Bug 或明确业务损失可以反超。

### 4.2 子任务

主任务使用飞书子任务表达可勾选执行步骤：

- 每个子任务为单一具体动作。
- 单项预计 15–45 分钟。
- 超过 45 分钟时继续拆分。
- 禁止使用“研究一下”“优化一下”“推进一下”等无法验收的描述。
- 子任务可以逐项完成，但不能因完成一个子任务而结束主任务。
- 所有必要子任务完成后，主任务才可进入完成或待验收。
- 固定节点以第一个未完成子任务作为下一步催办对象。

示例：

```text
主任务：完成并发布第 1 条 Codex 口播视频
├── 确定选题和核心观点
├── 完成 60 秒脚本第一版
├── 只修改一轮并定稿
├── 完成口播拍摄
├── 把原片交给剪辑
├── 验收成片
└── 发布并填写作品链接
```

### 4.3 状态来源

- 飞书任务及子任务状态是日常执行状态的正式依据。
- 知识库项目文件是项目目标、交付项和已验收进度的正式依据。
- 私聊中的“完成了”只能作为输入，不能绕过任务状态和成果验收。

## 5. 私聊输入处理

### 5.1 读取范围

每次节点运行时：

1. 从上一次成功检查点之后读取一对一私聊新消息。
2. 只接收配置的 N哥用户身份发送的消息。
3. 跳过机器人自己的消息。
4. 使用 `message_id` 去重。
5. 合并同一节点区间内的连续消息后统一分析。
6. 成功完成所有必要处理后才推进检查点。

### 5.2 消息分类

新增私聊必须被分类为以下一种或多种：

- 新任务。
- 临时想法或候选选题。
- 影响使用的故障。
- 会议或必须本人参与的事项。
- 执行卡点。
- 任务推迟原因。
- 成果或验收证据。
- 普通沟通。

普通沟通在没有任务影响、明确问题或沉淀价值时不回复。

### 5.3 四级处置

#### 立即插入

只有以下情况可打断或替换当前计划：

- 极享 OS 出现影响正常使用的 Bug。
- 客户、收入或正式交付正在发生明确损失。
- 事项必须由 N哥本人处理且有真实截止时间。
- 不处理会卡住多人继续工作。

#### 安排到今天

满足以下条件时进入当天剩余计划：

- 直接推进个人 IP 获客或极享 OS 上线。
- 可以在 120 分钟内产生明确成果。
- 当天 70% 容量仍有空间，或可以替换价值更低的任务。
- 不破坏两个核心项目最低投入，除非存在允许反超的 P0 情况。

#### 进入候选池

以下内容记录但不打断当天：

- 短视频选题。
- 产品功能灵感。
- 不影响当前使用的改进意见。
- 没有明确截止时间的想法。
- 尚未证明价值的事项。

#### 暂不安排

以下内容不占用 N哥核心时间：

- 与当前项目目标没有直接关系。
- 可以交给员工或 AI 处理。
- 只有主观紧急感，没有明确损失。
- 重复提出且没有新增信息。
- 当前阶段投入产出明显偏低。

暂不安排时说明原因、建议委派对象或重新评估条件。

### 5.4 优先级判断顺序

1. 是否正在造成业务损失。
2. 是否影响项目正常交付或系统使用。
3. 是否直接产生获客、收入或产品成果。
4. 是否必须由 N哥本人完成。
5. 是否有真实截止时间。
6. 是否卡住其他人。
7. 是否可以委派。
8. 是否只是一个待验证想法。

## 6. 各节点行为

### 6.1 08:00 今日任务分配

读取已确认周计划、项目最新进度、昨天未完成任务、待验收成果和 00:00–08:00 新私聊。生成或更新当天主任务与子任务，并发送一次机器人私聊总览。

总览只包含：今日必胜、任务数量、10:00 第一项任务、第一步动作和临时输入说明。详细执行内容保存在飞书任务中。

### 6.2 09:00 开工前校准

合并处理 08:00–09:00 新消息，判断是否需要更新 10:00 后的任务、时间或顺序。

- 有变化：更新飞书任务并发送一条合并调整说明。
- 无变化：保持安静。

### 6.3 12:00 上午进度核实

读取上午主任务、子任务和成果状态：

- 已完成：进入验收或确认完成，并明确 14:00 下一项。
- 部分完成：从第一个未完成子任务继续，必要时延续到下午。
- 完全无进度：将下一步缩小为 15 分钟动作，不重新讨论整套计划。
- 卡住：只追问一个具体卡点，收到答案后在下一节点处理或立即写入待处理状态。

12:00–14:00 不创建正式执行时间块。

### 6.4 15:00 下午启动检查

重点检查 14:00 任务是否启动、是否完成子任务、是否因低价值事项或会议被打断，以及是否反复修改同一内容。

- 正常推进：保持安静。
- 未开始：指定第一个 15 分钟最小动作和明确完成时间。
- 必须本人参加的临时事项：更新剩余计划。
- 非必要打断：保留核心任务，临时事项进入候选池。

### 6.5 18:00 白天成果验收

汇总完成任务、子任务、待验收成果、未开始任务、两个项目投入和新增私聊。晚间最多保留：

- 1 个核心主任务。
- 1 个不超过 30 分钟的整理或验收任务。

不把白天所有未完成事项原样堆到晚上。优先选择最接近交付的任务；个人 IP 当天没有成果时优先补个人 IP，极享 OS 不可用 Bug 除外。

### 6.6 21:00 晚间冲刺安排

合并处理 18:00–21:00 新消息，检查晚间任务：

- 已完成：要求补齐成果或进入验收，不再塞入大任务。
- 进行中：保持任务不变，明确最晚提交时间。
- 未开始：缩成唯一 15 分钟启动动作。

21:00 后不创建预计超过 120 分钟的新任务。执行最晚持续到 24:00。

### 6.7 24:00 每日复盘

读取最终任务和子任务状态，汇总：

- 主任务与子任务完成数量。
- 两个项目的实质成果。
- 待验收或未完成事项。
- 拖延、卡住和打断原因。
- 明日应该继续、缩小、取消、委派或重新评估的任务。

生成知识库每日复盘，并通过机器人私聊发送简洁摘要。24:00 后的新输入进入次日 08:00 处理区间。

## 7. 私聊输出规则

机器人只在以下情况发送消息：

- 08:00 今日总览。
- 新增私聊已经完成处理。
- 计划发生实质变化。
- 应该开始的任务没有进度。
- 需要说明卡点或推迟原因。
- 需要提交成果或人工验收。
- 18:00 晚间任务确定。
- 21:00 最后冲刺需要干预。
- 24:00 每日复盘。

同一节点所有新增消息和计划变化合并成一条回复。消息内容包含：读取数量、逐项处置结论、任务变化、当前唯一动作。没有新增消息、任务正常推进且无需调整时保持安静。

## 8. 催办与拖延处理

- 第一次无进度：指定 15 分钟最小动作。
- 连续两个节点无进度：要求说明一个具体卡点或推迟原因。
- 同一任务连续两次推迟：重新拆解、降低完成版本、委派或取消，不得原样顺延第三次。
- 只根据飞书任务和子任务的真实状态催办。
- 正在执行的任务不被普通重排打断。
- 所有时间、任务或子任务变化先更新飞书任务，再通过私聊解释原因。

## 9. Codex 自动化架构

### 9.1 执行模型

使用一个面向当前项目的 Codex 本地定时自动化，在七个固定时间唤起独立运行。每次运行：

```text
获取单实例运行锁
→ 识别当前管理节点
→ 读取上次成功检查点
→ 拉取私聊新消息
→ 拉取飞书主任务和子任务状态
→ 读取项目、周计划和当日计划
→ 执行节点策略
→ 创建或更新飞书任务
→ 必要时发送合并私聊
→ 写入知识库和运行状态
→ 提交检查点
→ 释放运行锁并结束
```

自动化调用项目内的确定性脚本完成飞书 API、SQLite 和 Markdown 操作；Codex 负责理解自由文本、价值判断和调度决策。脚本输出结构化结果，避免自动化依赖人工阅读终端日志。

### 9.2 运行状态

至少保存：

- 最后成功处理的消息检查点。
- 已处理 `message_id`。
- 每日各节点运行状态与版本。
- 飞书主任务、子任务与本地任务的对应关系。
- 已发送私聊回复的幂等键。
- 当日排程版本。
- 最后成功知识库写入版本。
- 当前单实例运行锁及过期时间。

### 9.3 幂等与并发

- 同一消息只处理一次。
- 同一逻辑任务不重复创建飞书任务或子任务。
- 同一日期同一节点可安全重试。
- 两个自动化实例重叠时只有取得锁的实例可修改状态。
- 状态写入和外部操作使用稳定幂等键。
- 只有全部必要操作成功后才推进消息检查点。

## 10. 异常与恢复

### 10.1 Mac 休眠、关机或断网

- 固定节点可能不能准时运行。
- 下一次成功运行补读所有未处理私聊和最新任务状态。
- 已过期的催办不补发，系统根据当前时间重新判断。
- 08:00 节点错过后，下一次节点先补建当日计划，再执行当前节点校准。
- 24:00 复盘错过后，在次日 08:00 排程前先补生成上一日复盘。

### 10.2 飞书失败

- 拉取失败：不推进消息检查点。
- 创建或更新任务失败：保存待重试操作，不发送“已安排”的错误结论。
- 私聊发送失败：进入发送队列并使用幂等键重试。
- 权限或身份错误：停止涉及写入的动作，并记录需要人工处理的明确错误。

### 10.3 知识库失败

- 项目 Markdown 继续使用哈希检查、临时文件和原子替换。
- 写入失败不破坏原项目文件。
- 飞书任务已完成但知识库未写回时，保留待协调状态，下次运行补写。

### 10.4 AI 判断失败

- 不猜测临时事项优先级或成果质量。
- 保留原输入并标记需要人工确认。
- 确定性任务状态和消息去重仍可继续处理。

## 11. 从当前系统切换

切换顺序：

1. 保留现有 SQLite、知识库项目文件、周计划和验收数据。
2. 增加一对一历史消息轮询与检查点能力。
3. 完成飞书主任务和子任务双向状态同步。
4. 增加固定节点执行器和静默输出策略。
5. 增加 Codex 自动化入口脚本和结构化执行摘要。
6. 停止当前 Terminal 长连接进程。
7. 创建并启用 Codex 固定时间自动化。
8. 通过真实一对一私聊完成端到端测试。
9. 确认没有长连接与轮询双重处理后正式切换。

切换不能在轮询能力验收前先停止现有服务。测试期间使用隔离的测试任务或明确测试标识，避免破坏正式项目进度。

## 12. 验收标准

### 12.1 私聊读取

- 能读取 N哥与机器人一对一私聊的新消息。
- 能跳过机器人消息和其他用户消息。
- 同一区间多条消息只生成一条合并回复。
- 同一消息重复轮询不会重复创建任务或回复。

### 12.2 飞书任务

- 08:00 能创建当天主任务和真实飞书子任务。
- 子任务完成状态能在下一节点被正确读取。
- 更新计划时修改原任务，不创建重复任务。
- 主任务和子任务均包含明确时间、动作与完成标准。

### 12.3 固定节点

- 七个节点分别执行规定职责。
- 无新增输入且进度正常时不发送多余消息。
- 错过节点后能按当前时间补处理，不补发过期催办。
- 21:00 后只保留一个核心冲刺任务，24:00 生成复盘。

### 12.4 调度与催办

- 临时事项按四级规则处理。
- 非 P0 新事项不会打断正在执行的核心任务。
- 首次无进度会生成 15 分钟最小动作。
- 连续推迟不会第三次原样顺延。

### 12.5 项目闭环

- 关键主任务完成后仍需成果验收。
- 验收通过后更新项目交付项、进度和变更记录。
- 飞书任务、私聊说明和知识库结果可以通过稳定标识追溯。

### 12.6 正式运行

- 不需要 Terminal 常驻。
- 不保持飞书 WebSocket 长连接。
- Codex 自动化运行记录可见。
- 当前长连接服务已停止，不会重复处理消息。

## 13. 第一版不包含

- 多员工、多项目群和团队权限管理。
- 群聊消息作为主要输入源。
- 秒级实时响应。
- 云端常驻服务。
- 自动读取日历或自动判断会议是否必须参加。
- 自动替代 N哥执行发布、付款、合同、报价、退款和删除。
- 对整个知识库进行无边界检索或修改。
