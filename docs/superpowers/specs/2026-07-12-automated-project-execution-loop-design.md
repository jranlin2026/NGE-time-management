# 自动化项目执行闭环设计

## 1. 目标

把现有飞书时间管理教练从“任务池排程器”升级为项目执行负责人。系统以知识库 Markdown 为唯一项目事实源，自动完成：

1. 读取项目目标、里程碑和真实进度。
2. 每周生成可确认的周计划。
3. 每天自动拆解并下发可验收任务。
4. 接收开始、卡住、推迟和完成反馈。
5. 对关键任务收集证据并验收。
6. 验收通过后自动更新项目进度和下一步。

目标闭环为：

`项目目标 → 周计划 → 每日任务 → 飞书执行 → 交付验收 → 项目进度更新`

## 2. 第一版范围

### 2.1 包含

- 结构化项目 Markdown 文件。
- 周日 22:00 自动生成周计划和飞书确认卡。
- 每日 08:00 自动生成并下发今日任务，不等待每日确认。
- 每日最多 5 个关键任务，只占可用时间的 70%。
- 个人 IP 和极享 OS 的任务配额、最低投入和受保护时间窗。
- 普通任务直接完成，关键任务先进入待验收。
- 验收通过后自动更新里程碑、项目进度和变更记录。
- 现有开始、卡住、推迟、10+10 分钟催办、动态重排和 24:00 复盘流程。

### 2.2 不包含

- Codex 自动代替用户执行写文档、改代码等工作。
- 员工任务分配、团队权限和多人进度管理。
- 日历或会议系统自动读取。
- 自动付款、发布、删除、对外承诺等高风险操作。
- 对整个知识库进行无边界的 AI 搜索和修改。

## 3. 唯一事实源与目录结构

知识库目录为：

```text
08_时间管理大师/
├── 项目/
│   ├── 个人IP.md
│   └── 极享OS.md
├── 周计划/
│   └── YYYY-Www.md
├── 每日计划/
│   └── YYYY-MM-DD.md
├── 每日复盘/
│   └── YYYY-MM-DD.md
└── 项目变更记录/
    └── YYYY-MM-DDTHH-mm-ss-项目ID.md
```

规则：

- `项目/*.md` 是项目目标、里程碑和正式进度的唯一事实源。
- `周计划/*.md` 中状态为 `confirmed` 的文件是当周执行依据。
- SQLite 只保存运行状态、飞书事件、提醒、排程版本、验收过程和发送队列。
- SQLite 不得反向覆盖更新版本更高的 Markdown 项目内容。
- 系统只写入 Markdown 中明确标记的自动管理区域，不修改自由笔记。

## 4. 项目文件格式

每个项目一个独立文件，使用稳定项目 ID，不以文件名作为外键。

```markdown
---
project_id: personal-ip
name: 个人IP
status: active
priority: 1
updated_at: 2026-07-12T08:00:00+08:00
---

# 个人IP

## 项目目标

通过国内与海外短视频持续获取企业老板和实体店老板线索。

<!-- time-manager:managed:start -->
## 当前阶段

内容冷启动

## 里程碑

| milestone_id | 名称 | 截止时间 | 项目权重 | 状态 |
| --- | --- | --- | ---: | --- |
| ip-content-validation | 验证内容方向 | 2026-07-31 | 100 | active |

## 里程碑交付项

| deliverable_id | milestone_id | 交付项 | 里程碑权重 | 状态 | 验收证据 |
| --- | --- | --- | ---: | --- | --- |
| ip-video-01 | ip-content-validation | 发布第 1 条短视频 | 10 | pending | |

## 当前风险

- 脚本文案容易反复修改，迟迟不拍摄。

## 下一步候选

- 完成第 1 条口播脚本第一版。

## 最近一次实质成果

尚无。
<!-- time-manager:managed:end -->

## 自由笔记

系统不得修改本节。
```

约束：

- 同一项目的里程碑“项目权重”合计必须为 100。
- 同一里程碑的交付项“里程碑权重”合计必须为 100。
- 每个每日关键任务必须绑定 `project_id`、`milestone_id` 和 `deliverable_id`。
- 项目文件缺失或格式不合法时，该项目停止自动更新，并在飞书报告具体错误。
- 首次升级时，如果项目文件不存在，系统生成 `draft` 模板并要求确认；未确认模板不得成为自动计划依据。

## 5. 进度计算

进度只根据已验收交付项计算，不允许 AI 直接填写主观百分比。

```text
里程碑进度 = 已验收交付项权重之和
项目进度 = Σ(里程碑进度 × 里程碑项目权重 ÷ 100)
```

所有权重按百分制计算，结果限制在 0–100%。周计划可以提出新增或调整交付项，但只有用户确认周计划后才写入项目文件。每日重排只能选择已确认交付项，不得偷偷扩大项目范围。

## 6. 系统模块

### 6.1 项目仓库

职责：读取、校验和安全更新 `项目/*.md`。

- 返回结构化项目、里程碑、交付项和版本哈希。
- 写入前再次检查文件哈希；文件已被人工修改时重新读取并重新计算。
- 先写临时文件，再原子替换。
- 每次更新前生成项目变更记录。

### 6.2 周计划生成器

职责：从项目事实生成一周可验收结果。

- 每周日 22:00 运行。
- 读取所有 `active` 项目、当前里程碑、风险、最近成果和未验收交付项。
- 每个周任务必须绑定现有交付项，并包含交付物、完成标准、预计时长和建议日期。
- 输出 `周计划/YYYY-Www.md`，初始状态为 `draft`。
- 向飞书发送“确认计划”和“需要调整”按钮。

### 6.3 每日任务生成器

职责：从已确认周计划、前日结果和未完成任务中生成当天安排。

- 每天 08:00 运行。
- 不等待每日确认，直接创建飞书任务并发送群内作战卡。
- 单日最多 5 个不同关键任务。
- 任务池充足时，个人 IP 至少 2 项、极享 OS 至少 2 项。
- 两个项目每天最低各保留 120 分钟投入。
- 个人 IP 使用 10:00–12:00、14:00–16:00。
- 极享 OS 使用 10:00–12:00、14:00–24:00。
- 12:00–14:00 不安排任务。
- 单个时间块不超过 120 分钟。
- 总排程分钟数不得超过所有可用时间窗分钟数的 70%；剩余 30% 留给会议、沟通、休息和突发事项。
- 当当天可用时间不足以同时满足 70% 容量上限和两个项目各 120 分钟最低投入时，容量上限优先，系统在飞书报告未满足的项目投入，不得超额排程。
- 个人 IP 默认优先；只有明确标记为“影响系统使用的 Bug”的极享 OS 任务可以反超。

### 6.4 交付验收器

职责：判断关键任务是否真正完成。

- 普通任务点击完成后直接进入 `done`。
- 关键任务点击完成后进入 `pending_acceptance`，不增加项目进度。
- 绑定项目交付项的任务默认 `requires_evidence=true`，属于关键任务；不绑定项目交付项的行政或运行任务默认 `requires_evidence=false`，属于普通任务。周计划可以显式提高验收要求，但不得把项目交付任务降为普通任务。
- 飞书追问链接、截图、视频、页面地址或文字结果。
- 数量、URL、文件存在性等明确标准优先使用确定性规则检查。
- 语义质量标准由 AI 返回 `accepted`、`rejected` 或 `needs_user_confirmation`，并附具体依据。
- AI 不可用、证据无法读取或结论不明确时，一律进入 `needs_user_confirmation`，不得猜测通过。
- 验收不通过时说明缺口，并生成绑定同一交付项的返工任务。

### 6.5 进度写回器

职责：在验收通过后更新项目事实源。

- 将交付项状态更新为 `accepted`，写入证据引用和验收时间。
- 重新计算里程碑进度和项目进度。
- 更新最近一次实质成果和下一步候选。
- 写入包含变化前后值、任务、证据和原因的变更记录。
- 飞书发送“项目推进了什么、当前进度、下一步是什么”。

## 7. 周计划流程

1. 周日 22:00 读取项目事实源。
2. 生成周计划草稿和飞书确认卡。
3. 用户点击“确认计划”后，将周计划状态改为 `confirmed` 并生效。
4. 用户点击“需要调整”后，系统要求一条调整说明，重新生成草稿。
5. 周一 08:00仍未确认时，不采纳草稿中的新方向；系统沿用上一次已确认周计划和当前最高优先项目执行，并再次提醒确认。
6. 同一周同一版本的确认事件只处理一次。

## 8. 每日执行流程

1. 08:00 从已确认周计划生成今日安排。
2. 到任务时间发送“现在只做这一件”卡片。
3. 同一时间只允许一个任务为 `doing`。
4. 点击“开始”后进入 `doing`，原卡片更新并取消无响应提醒。
5. 点击“完成”后，根据任务类型进入 `done` 或 `pending_acceptance`。
6. 点击“卡住”后记录卡点，生成 15 分钟最小动作并重排后续时间。
7. 点击“推迟 30 分钟”后必须补充原因；到期恢复任务并重排。
8. 开始时间后 10 分钟无反馈时群内 @ 提醒。
9. 再过 10 分钟无反馈时记录拖延、缩小任务并重排。
10. 临时事项只有被明确标记“必须本人处理”时才能打断当前任务。
11. 12:00 和 18:00根据最新状态重排；进行中的任务不被重排打断。
12. 24:00生成复盘并写入知识库。

## 9. 状态与运行数据

任务状态扩展为：

```text
inbox → ready → scheduled → doing → pending_acceptance → done
                              ↘ blocked
                              ↘ deferred → ready
```

新增运行数据：

- `weekly_plans`：周次、Markdown 路径、版本、状态和确认事件。
- `task_acceptances`：任务、交付项、证据引用、验收结果、说明和时间。
- `project_sync_state`：项目文件路径、最后读取哈希、最后写入版本和错误。

正式项目内容不存入这些表作为事实源；表只用于幂等、恢复、查询和飞书交互。

## 10. 异常处理

- 项目文件格式错误：停止该项目自动更新，飞书报告字段和位置，不覆盖文件。
- 周计划未确认：继续上一次已确认方向，不让每日执行停摆。
- 验收证据不足：保持待验收，不增加进度。
- 飞书事件重复：根据事件 ID 幂等处理。
- 飞书发送失败：进入现有发送队列，最多重试 8 次。
- AI 分析失败：保留原任务，使用规则生成最小下一步，并标记待重新分析。
- 知识库并发修改：写入前哈希冲突时放弃本次写入，重新读取后再计算。
- 知识库写入失败：保留临时文件和错误事件，项目原文件不受影响。
- Mac 休眠或关机：恢复后废弃过期提醒，根据当前时间重排。
- 当前任务正在执行：任何重排只改变后续任务。

## 11. 测试策略

### 11.1 项目文件测试

- 正确解析项目、里程碑和交付项。
- 拒绝重复 ID、权重不为 100、缺少管理区块和非法状态。
- 只更新管理区块，自由笔记保持逐字不变。
- 文件哈希冲突时不覆盖人工修改。

### 11.2 周计划测试

- 周日生成 `draft` 周计划和确认卡。
- 确认后状态变为 `confirmed`。
- 未确认时只沿用上周方向，不引入草稿新交付项。
- 重复确认不产生重复任务或重复进度。

### 11.3 每日排程测试

- 每日最多 5 件，项目任务池充足时各至少 2 件。
- 两个项目各保留至少 120 分钟。
- 任务不越过项目时间窗和 12:00–14:00 禁排时间。
- 总排程不超过可用分钟数的 70%。
- 影响使用的极享 OS Bug 可以反超个人 IP，普通 OS 任务不能。

### 11.4 验收与进度测试

- 普通任务直接完成，关键任务进入待验收。
- 证据不足不改变项目进度。
- 验收通过只增加对应交付项权重一次。
- 验收失败生成返工任务。
- 项目进度计算与变更记录一致。

### 11.5 飞书端到端测试

- 周计划生成、调整、确认和未确认回退。
- 今日计划下发、开始、卡住、推迟和完成。
- 提交结果、验收通过、项目 Markdown 写回和进度通知。
- 服务重启后不重复处理既有事件。

## 12. 第一版验收标准

1. 给定两个合法项目文件，周日 22:00 自动生成一份周计划草稿并发送飞书确认卡。
2. 确认周计划后，周一 08:00 自动生成最多 5 件任务，满足 70% 容量、项目配额、最低投入和时间窗。
3. 一个关键任务完成后进入待验收；提交有效证据并验收通过后，对应交付项只完成一次。
4. 项目 Markdown 中的里程碑、进度、最近成果和下一步自动更新，自由笔记不变，并生成变更记录。
5. 卡住、推迟、无响应和临时打断仍能触发现有重排与教练流程。
6. Mac 或服务重启后，周计划、待验收任务和项目同步状态可以恢复，飞书事件不会重复执行。
