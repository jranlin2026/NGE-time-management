# N哥个人时间管理：飞书任务与私聊执行体验重构设计

状态：设计已确认，待实施计划  
日期：2026-07-12  
适用范围：只管理 N哥本人，不管理团队成员任务  
批准依据：2026-07-12 用户确认采用“结果主任务＋带时间子任务”，并确认七节点动态调整逻辑

## 1. 一句话结论

飞书只保存 N哥本人可验收的结果主任务和带时间子任务；机器人私聊将同一份计划渲染成可直接照着执行的全天日程；系统只保留一条飞书写入链路，任何重跑只能更新原任务，不能再次创建。

## 2. 当前问题与证据

### 2.1 飞书出现重复任务

当前一次 08:00 排程同时触发两条写入链路：

1. `manager-service.mjs` 把旧式 `feishu_task_create` 写入 outbox。
2. `checkpoint-runner.mjs` 调用 `feishu-task-sync.mjs` 创建 managed 主任务与子任务。
3. runner 随后 flush outbox，旧式全天任务又被创建一次。

真实飞书任务清单现有 14 个顶层任务：

- 4 个当前有效 managed 任务：有准确时间、各有 3 个子任务。
- 5 个旧 outbox 重复任务：全天、无子任务、未进入 `feishu_task_links`。
- 5 个已完成历史任务：保留作为历史记录。

本地任务标题并没有重复，因此不能通过标题去重。必须先关闭双写，再按远端 GUID、outbox 来源和 `feishu_task_links` 归属精确清理。

### 2.2 私聊无法指导执行

当前 08:00 私聊只输出：今日必胜、任务数量、第一步、临时输入说明。原因是 `checkpoint-policy.mjs` 的 `renderDispatch()` 只生成四行文本，没有消费排程中已经存在的时间块、工作内容和完成标准。

原规格也把详细内容留给飞书任务，但当前飞书主任务描述取用了不存在的 `localTask.description`，子任务描述固定为空。因此用户在私聊和飞书任务中都看不到足够的执行详情。

### 2.3 团队结果被错误写成个人任务

“完成公域直播30单”会让人理解为 N哥本人直播 14 小时。实际情况是三名主播执行 10:00–24:00 的直播，N哥只需要在必要节点检查、决策和复盘。

本项目从本设计起只管理 N哥本人的时间。团队工作只能作为背景、依赖或验收对象，不能占用 N哥的个人执行时长。

## 3. 方法论与产品原则

本设计采用“飞书保存结果，私聊呈现执行”的混合模型。

### 3.1 一个今日必胜，正常 3–4 个承诺结果

- 每天必须有 1 个今日必胜。
- 正常安排 3–4 个可验收结果，最多 5 个。
- 不按数量凑任务；最终数量必须受可用时间约束。
- 初始只占用约 70%–80% 的可用时间，其余用于沟通、切换、临时决策和估时偏差。该比例是试运行参数，不宣称为固定科学定律。

### 3.2 明确何时、何地、如何行动

实施意图研究支持把目标写成明确的“何时、何地、如何行动”；94 项独立检验的元分析报告了中到大的总体效应。产品落地为：每个子任务必须有绝对时间、动作和完成标准。  
来源：[Gollwitzer 与 Sheeran 元分析](https://doi.org/10.1016/S0065-2601%2806%2938002-1)

### 3.3 固定反馈节点并记录结果

进度监测能促进目标达成，且记录或向他人报告时效果更强。产品落地为：12:00、15:00、18:00、21:00、24:00 读取实际完成状态并在必要时重排。  
来源：[Harkin 等人进度监测元分析](https://eprints.whiterose.ac.uk/id/eprint/87431/1/bul%20harkin%20raw%20final.pdf)

### 3.4 减少任务切换并保留缓冲

未完成任务切换会产生注意力残留，影响下一任务表现。产品落地为：保护正在进行的任务、连续安排同类动作、临时事项默认进入后续空档或候选池。  
来源：[Leroy 关于注意力残留的研究](https://www.sciencedirect.com/science/article/abs/pii/S0749597809000399)

### 3.5 记录预计与实际用时

计划谬误会使人低估完成时间；参考相似任务的历史结果可以改善估时。产品落地为：任务保留预计用时，复盘记录实际用时，用一周真实数据校准后续排程。  
来源：[Buehler、Griffin 与 Ross](https://web.mit.edu/curhan/www/docs/Articles/biases/67_J_Personality_and_Social_Psychology_366%2C_1994.pdf)

## 4. 任务信息模型

### 4.1 主任务代表一个可验收结果

标题格式：

```text
项目｜动作 + 可验收结果
```

规则：

- 日期不重复写进标题，日期由开始时间和截止时间表达。
- 标题不得使用“推进、处理、优化、研究一下”等无法验收的动词。
- 团队执行结果必须改写成 N哥本人的检查、决策、验收或复盘动作。
- 同一个当天结果只能有一个主任务。

主任务最小字段：

- 稳定业务键：`workDate + project + resultKey`。
- 项目。
- 结果标题。
- 目的。
- 第一动作。
- 完成标准。
- 交付物或证据要求。
- 预计总用时与实际用时。
- 优先级。
- 当前状态。

### 4.2 子任务代表一个可立即执行的时间块

子任务标题格式：

```text
HH:mm–HH:mm｜动作 + 阶段产出
```

每个子任务保存：

- `checkpoint_index`。
- 开始时间与结束时间。
- 预计分钟数。
- 工作内容。
- 可观察的完成标准。
- 证据或反馈方式。
- 完成状态。

单个子任务原则上为 15–45 分钟；超过 45 分钟时拆成连续阶段。等待剪辑、员工操作或主播直播不计入 N哥个人专注时长。

### 4.3 排程数据结构

`schedule_blocks` 增加可空的 `checkpoint_index`：

- 有 `checkpoint_index`：表示一个具体子任务时间块。
- 无 `checkpoint_index`：兼容旧数据和无法继续拆分的临时任务。
- 主任务开始时间取其最早子任务开始时间。
- 主任务截止时间取其最晚子任务结束时间。
- 一个主任务允许出现多个不连续时间块，例如上午拍摄、晚上验收发布。

同一份 schedule 同时作为飞书任务同步和私聊渲染的事实源，禁止分别计算两套时间。

## 5. 明日任务重写示例

### 5.1 个人 IP

主任务：`个人IP｜交付3条可剪辑原片并发布3条视频`

- `10:15–10:35｜确定3个选题与开头钩子`
- `10:35–11:15｜完成3条口播提纲`
- `11:15–12:00｜拍摄3条可剪辑原片`
- `18:30–19:00｜验收成片并发布3条视频`

完成标准：3条原片音画可用并交给剪辑；3条成片完成标题、封面和文案检查后发布；保存发布链接或截图。

### 5.2 极享 OS

主任务：`极享OS｜验收线索模块并完成1名员工真实迁移`

- `14:00–14:30｜确认线索字段、权限与操作流程`
- `14:30–15:15｜完成新增、分配、跟进和查询抽测`
- `15:15–16:00｜让1名员工完成真实线索操作并记录问题`

完成标准：员工能独立完成主流程；阻断问题为零；非阻断问题进入问题清单并明确负责人和时间。

### 5.3 公域直播

主任务：`公域直播｜完成开播检查与30单结果复盘`

- `10:00–10:15｜确认三主播排班、货盘和成交口径`
- `18:00–18:15｜检查中场订单数据并只处理关键阻塞`
- `23:30–24:00｜记录实际订单、差距和明日唯一纠偏动作`

三名主播 10:00–24:00 的执行时间只写在背景说明中，不计入 N哥个人时间。

### 5.4 私域直播

主任务：`私域直播｜确定本周首场方案并启动邀约`

- `20:00–20:20｜确定主题、目标人群和成交产品`
- `20:20–20:40｜确认海报文案、邀约名单和发送节奏`
- `20:40–21:00｜明确直播分工并启动第一轮邀约`

完成标准：主题、邀约对象、成交产品、文案、发送节奏和负责人全部明确，第一轮邀约已启动。

## 6. 08:00 私聊执行令

08:00 每天必发一条完整执行令，必须回答：今天拿到什么结果、几点做什么、每块产出什么、第一步是什么、怎样算完成、何时反馈、延误怎么办。

标准结构：

```text
【7月13日执行令｜今天只完成4个结果】

今日胜利条件：
1. 发布3条个人IP视频
2. 极享OS线索模块由1名员工完成真实操作
3. 公域直播完成开播检查，并输出30单结果与纠偏动作
4. 完成7月15日私域直播方案并启动邀约

【时间安排】

10:00–10:15｜检查公域直播启动
工作内容：确认3名主播排班、货盘、设备和成交口径。
完成标准：主播均已进入执行，30单统计口径明确。

10:15–12:00｜交付3条个人IP原片
第一步：确定3个选题和开头钩子。
完成标准：3条可剪辑原片已交付剪辑人员。

12:00–14:00｜午休，不安排任务

14:00–16:00｜验收极享OS线索模块
完成标准：1名员工完成真实线索新增、分配、跟进和查询。

16:00–16:30｜缓冲与必须本人决策的团队阻塞

18:00–18:15｜检查直播中场数据
18:30–19:00｜验收成片并发布3条视频
20:00–21:00｜确定7月15日私域直播方案并启动邀约
23:30–24:00｜复盘公域直播30单结果

【今天不做】
不研究出海，不新增项目，不反复修改已经可发布的文案。

【反馈规则】
完成：在飞书点对应子任务。
卡住：回复“卡住：任务名｜原因”。
推迟：回复“推迟：任务名｜原因｜新的完成时间”。
```

私聊是同一批飞书任务的执行视图，不是第二套任务。

## 7. 七个固定节点的动态调整

| 节点 | 职责 | 发送规则 |
|---|---|---|
| 08:00 | 生成完整全天执行令，同步主任务和带时间子任务 | 每天必发 |
| 09:00 | 读取 08:00 后新增事项，校验计划 | 只有新增或变化才发 |
| 12:00 | 核对上午成果，提前则拉入一个高价值动作，滞后则缩小范围 | 有变化或异常才发 |
| 15:00 | 检查下午启动，处理午间新增事项 | 有变化、未启动或卡点才发 |
| 18:00 | 白天收口，明确晚上保留和取消的事项 | 有晚间任务或变化才发 |
| 21:00 | 最终冲刺，只保留一个最接近交付的结果 | 有未完成关键交付才发 |
| 24:00 | 复盘完成率、延期原因、直播数据与明日候选 | 每天必发 |

重排规则：

1. 保护已经开始的任务，只调整后面的时间块。
2. 提前完成后只补一个高价值动作，不重新塞满空闲时间。
3. 只有极享 OS 不可用、正在造成业务损失、必须由 N哥今天决定、或阻塞多人时才能立即插队。
4. 普通新增事项进入后续空档或候选池。
5. 调整消息只包含：事实、变化、最新时间、现在唯一动作、反馈截止时间。
6. 无变化保持安静，不重复发送整份日程。

差量消息示例：

```text
【12:00计划调整】
变化：个人IP拍摄提前40分钟完成。
保持不变：14:00–16:00 极享OS线索模块验收。
新增安排：如成片已返回，18:30的发布验收提前到11:20–11:50；否则保留为空闲缓冲。
现在只做：把3条原片和标题交给剪辑人员。
反馈截止：12:00前完成飞书子任务勾选。
```

## 8. 唯一飞书同步链路

### 8.1 单写者原则

固定节点模式只允许 `feishu-task-sync` 根据 `feishu_task_links` 写入飞书任务。

- `task_dm` / checkpoint 排程不得再入队旧 `feishu_task_create`。
- managed 主任务和子任务统一使用稳定业务键、managed marker 和 client token。
- 本地完成状态只更新 `feishu_task_links` 对应的 managed GUID。
- 旧 `settings.feishu_task_guid:*` 不再作为固定节点完成同步依据。
- 同一时间只能启用 Codex 固定节点自动化；旧 WebSocket/launchd manager 必须卸载，不能仅检查当前是否有 PID。

### 8.2 幂等与崩溃恢复

- 重复运行同一节点只能更新相同父任务和子任务。
- 父任务创建成功但链接落库失败时，重试必须通过稳定 client token 或 managed marker 收敛到原任务。
- 如果飞书列表接口缺少恢复字段，则创建后必须通过详情接口或稳定 token 查询确认，不能直接再次创建。
- 私聊发送必须在任务同步成功之后；同步失败时不得宣称安排完成。

失败消息：

```text
计划已经生成，但飞书任务同步失败。
当前不要按旧任务执行；系统将在下一节点重试，并在同步完成后发送最新版执行令。
```

## 9. 线上重复任务清理

必须按以下顺序实施：

1. 暂停 Codex 自动化。
2. 卸载仍处于 KeepAlive/RunAtLoad 的旧 LaunchAgent。
3. 部署并验证单一 managed 同步链路。
4. 使用 SQLite backup API 生成可打开的数据库备份。
5. 回读飞书任务清单，按远端 GUID、sent `feishu_task_create` 来源、未进入 `feishu_task_links`、全天、0 子任务五项联合核验。
6. 精确删除 5 个旧 outbox 重复任务；禁止只按标题删除。
7. 保留 5 个已完成历史任务。
8. 将当前两个人 IP 父任务合并为一个结果主任务，更新其 timed checkpoints。
9. 重写极享 OS 与公域直播任务，新增私域直播个人准备任务。
10. 重新同步并连续运行两次。
11. 验证第二次父任务新增数和子任务新增数均为 0。
12. 恢复 Codex 自动化。

清理完成后的预期顶层任务：5 个已完成历史任务，加当天 4 个清晰的待办结果任务；无旧全天重复任务。

## 10. 模块边界

### 10.1 `daily-execution-brief.mjs`

纯函数模块，输入任务、checkpoint 时间块、时区、固定反馈节点和计划差异，输出：

- 08:00 完整执行令文本。
- 09/12/15/18/21 差量消息文本。
- 纯文本降级内容。

它不读取数据库、不调用飞书、不修改排程。

### 10.2 调度引擎

只决定 N哥本人哪些 checkpoint 在什么时间执行，保留缓冲并保护进行中时间块。它不渲染用户文案，也不调用飞书。

### 10.3 飞书任务同步器

只把同一份执行计划同步为一个结果父任务和若干带时间子任务，负责幂等、采用和完成状态回拉。它不决定优先级。

### 10.4 检查点策略

读取新增消息、远端完成状态和当前计划，决定是否重排；调用执行简报渲染器生成完整或差量消息。

## 11. 验收标准

### 11.1 任务清晰度

- 飞书当天待办只有 3–5 个结果主任务。
- 同一业务结果只出现一次。
- 每个主任务标题包含项目、动作和可验收结果，不重复日期。
- 每个子任务包含绝对时间、具体动作和阶段产出。
- 团队执行时间不计入 N哥个人计划。

### 11.2 私聊可执行性

- 08:00 私聊包含所有实际时间块、工作内容、第一步、完成标准、不做清单和反馈格式。
- UTC 时间必须正确显示为 Asia/Shanghai 本地时间。
- 同一主任务跨多个时间块时标题只出现一次，时间块完整呈现。
- 09:00 无变化保持安静。
- 12/15/18/21 的变化消息包含事实、调整、新时间、唯一动作和反馈截止时间。

### 11.3 同步与去重

- 一次检查点只能创建一个 managed 父任务和预期数量的 managed 子任务。
- 连续运行两次后，第二次新增父任务和子任务均为 0。
- 不再产生固定节点旧式 `feishu_task_create` outbox。
- 删除重复任务后，飞书顶层任务结构与预期一致。
- 旧 LaunchAgent 不再处于 loaded、KeepAlive 或 spawn scheduled 状态。

### 11.4 动态重排

- 提前完成后只引入一个高价值动作，并保留缓冲。
- 新增普通事项不会打断进行中任务。
- 合法 P0 能插队，并明确说明被移动的任务和新时间。
- 延误超过 15 分钟时生成缩小范围或新时间，不复制一条新任务。
- 预计用时和实际用时进入 24:00 复盘。

## 12. 非目标

- 不管理主播、剪辑、员工或其他团队成员的个人任务。
- 不建设团队任务清单或员工绩效系统。
- 不接入云端常驻服务。
- 不用更多提醒次数代替任务清晰度。
- 不删除历史已完成任务。
- 不在本次重构中扩展日历、CRM、邮件或其他外部系统。
