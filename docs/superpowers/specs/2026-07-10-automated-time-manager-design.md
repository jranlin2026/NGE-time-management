# 自动化时间管理负责人设计

日期：2026-07-10  
状态：已通过交互式设计确认，待用户审阅书面规格

## 1. 目标

把现有“时间管理大师”MVP升级为在当前 Mac 上长期运行的个人时间管理负责人。用户只通过飞书输入任务和反馈状态；系统负责补全信息、排序、安排时间、主动提醒、处理拖延、动态重排，并在晚上生成复盘。

第一版只管理用户本人，不管理团队成员，不接入日历、CRM 或其他任务来源。

## 2. 已确认的产品决策

- 飞书是唯一任务入口和交互界面。
- Mac 本地任务库是主数据源，飞书任务与消息是同步界面。
- 系统拥有日内调度权，可以不经确认直接重排任务。
- 采用强管理节奏：早晨计划、任务开始提醒、两次超时处理、午间校准、白天收口和晚间复盘。
- 卡片按钮和自然语言文字都可以更新任务状态。
- AI 与规则采用混合模式：本机 Codex 负责理解和拆解，规则引擎负责最终时间与边界决策。
- 不使用独立 OpenAI API；通过本机 `codex exec` 非交互调用 Codex。
- 系统由 macOS `launchd` 守护，开机启动并在异常退出后自动拉起。
- 第一版完成标准包含自动拆任务、超时重排和晚间复盘；周报和长期效率分析不在第一版范围。

## 3. 使用场景

### 3.1 新增任务

用户可以在飞书里直接说：

> 这周要把 15 条个人 IP 视频拍完。

系统先可靠保存原始消息，再调用 Codex 生成结构化任务信息，包括项目、标题、截止时间、重要性、紧急性、预计耗时、下一步动作和完成标准。信息缺失时由系统合理补全；只有存在明显冲突或高风险时才追问。

### 3.2 执行任务

系统在时间块开始时只展示当前任务，并提供：

- 开始
- 完成
- 卡住
- 推迟 30 分钟

用户也可以发送“开始：任务名”“完成：任务名”或“卡住：任务名 + 原因”。

### 3.3 拖延干预

任务开始提醒发出 15 分钟后仍无反馈，系统第一次追问。再过 15 分钟仍无反馈，系统：

1. 记录一次拖延事件；
2. 把当前任务缩小为 15 分钟最小动作；
3. 重排当天剩余时间块；
4. 通过飞书说明修改内容和原因。

系统不会无限每 15 分钟轰炸，也不会因为无响应静默放弃任务。

### 3.4 卡住

“卡住”不会关闭原任务。系统保存具体卡点，调用 Codex 建议一个 15 分钟最小动作，由规则引擎确认其不超出当前时间块后下发。完成最小动作后，原任务回到进行中或被重新安排。

### 3.5 晚间复盘

晚间任务结束后，系统生成：

- 今日关键任务完成数；
- 已完成、延期和仍在进行中的事项；
- 拖延次数与具体卡点；
- 当天发生过的重排及原因；
- 明日候选任务和一条可执行建议。

## 4. 每日管理节奏

| 时间 | 系统行为 |
| --- | --- |
| 08:30 | 下发 1–3 个今日关键任务、时间块、下一步、完成标准和排序原因 |
| 10:00 | 启动第一个任务；无响应时执行两级超时处理 |
| 12:00 | 根据完成、进行中、卡住和临时任务重排下午 |
| 14:00–18:00 | 按时间块推进；完成后进入下一项，卡住时拆小 |
| 18:00 | 白天收口；最多保留一个未完成关键任务进入晚上，其余延期 |
| 20:00–22:00 | 默认晚间工作时段；结束后生成复盘 |

时间点必须保存在设置中，而不是写死在业务逻辑里，以便后续调整。

## 5. 系统架构

采用一个由 `launchd` 守护的 Node.js 常驻服务。服务内部划分为边界清晰的模块，而不是多个互相竞争的定时脚本。

### 5.1 飞书适配器

职责：

- 通过现有 WebSocket 长连接接收消息；
- 接收卡片按钮事件；
- 发送普通消息和互动卡片；
- 创建或更新飞书任务；
- 使用飞书消息 ID 和卡片事件 ID 做幂等去重。

该模块只负责传输和格式转换，不决定任务优先级。

### 5.2 任务接收服务

职责：

- 保存原始输入；
- 将输入分类为新增、开始、完成、卡住、推迟、计划查询或非任务对话；
- 对新任务发起 Codex 结构化分析；
- 在 Codex 不可用时应用默认值并标记“待重新分析”。

### 5.3 Codex 分析适配器

通过本机 `codex exec` 调用，使用 JSON Schema 限定返回格式。Codex 只返回建议，不直接操作 SQLite、飞书或项目文件。

输入限制为完成判断所需的最小上下文：原始消息、当前项目摘要、可用工作时间和少量候选任务。输出包括：

- 意图分类；
- 结构化任务字段；
- 建议的下一步动作；
- 建议的完成标准；
- 对卡点的 15 分钟最小动作；
- 晚间复盘文本草稿。

调用必须设置超时、输出格式校验和错误日志。失败时使用规则默认值，不能阻塞任务入池。

### 5.4 调度引擎

职责：

- 从未完成任务中选择当天最多 1–3 个关键任务；
- 根据可用时间、预计耗时和任务依赖生成时间块；
- 处理临时任务、完成、卡住、推迟和无响应事件；
- 每次状态变化后重新计算尚未开始的时间块；
- 保存每次调度的原因，供飞书解释和晚间复盘使用。

第一版沿用并扩展现有优先级因素：四象限、截止时间、S/A/B/C 重要性、紧急性和拖延次数。规则引擎是最终决策者，Codex 只能提供结构化字段和建议分数。

对于 2026-07-10 至 2026-07-15 的当前冲刺，个人 IP 项目具有冲突优先权。该规则应作为有起止日期的项目优先级配置保存，而不是永久写死。

### 5.5 提醒与恢复引擎

职责：

- 生成计划提醒、开始提醒、超时检查、午间校准、18:00 收口和晚间复盘任务；
- 读取数据库决定下一次需要执行的动作；
- 在服务重启或 Mac 唤醒时重新计算，而不是依赖内存定时器；
- 跳过已经失去意义的旧提醒，改为发送一张“恢复后的新计划”。

### 5.6 飞书发件箱

所有待发送消息先与业务状态在同一个 SQLite 事务中写入发件箱，再由后台发送器投递飞书。发送失败采用带上限的指数退避重试。成功后记录飞书消息 ID；重复执行不会产生第二份业务结果。

### 5.7 Markdown 导出

SQLite 是主数据源。系统继续生成可读的每日 Markdown 计划和复盘，用于 FounderOS 或人工查看，但 Markdown 不参与反向覆盖数据库。

## 6. 数据模型

### 6.1 `tasks`

保存任务本身：ID、标题、项目、原始输入、重要性、紧急性、四象限、截止时间、预计分钟、下一步、完成标准、状态、卡点、拖延次数、来源消息 ID、分析状态和时间戳。

状态限定为：

- `inbox`
- `ready`
- `scheduled`
- `doing`
- `blocked`
- `done`
- `deferred`
- `cancelled`

### 6.2 `schedule_blocks`

保存每天实际安排：任务 ID、开始和结束时间、顺序、状态、生成原因、版本号和被替换版本。重排创建新版本并保留旧版本用于追踪。

### 6.3 `task_events`

使用只追加事件记录：新增、分析完成、安排、开始、完成、卡住、推迟、拖延、重排和取消。每条外部事件都有幂等键。

### 6.4 `reminders`

保存提醒种类、计划触发时间、实际触发时间、状态、关联任务、尝试次数和失效时间。

### 6.5 `outbox`

保存待发送的飞书文本、卡片或任务同步操作、幂等键、重试次数、下一次重试时间和最终结果。

### 6.6 `daily_reviews` 与 `settings`

`daily_reviews` 保存每日汇总和明日候选；`settings` 保存时区、工作时段、提醒间隔、每日关键任务上限和临时项目优先级。

## 7. 状态与同步规则

1. 外部消息先按事件 ID 去重。
2. 任务状态、事件记录和待发消息在一个 SQLite 事务中提交。
3. 飞书发送和任务同步在事务后异步完成。
4. 本地状态与飞书展示冲突时，以本地事件日志为准，并重新同步飞书。
5. “完成”是终态，除非用户明确发送“恢复：任务名”。
6. “延期”保留原任务和历史，下一次评分时增加拖延权重。
7. 同一时间最多存在一个 `doing` 任务，系统需要明确结束、暂停或重排当前任务后才能启动下一项。

## 8. 故障处理

### 8.1 Codex 超时或输出无效

- 任务立即以默认值进入收件箱，不丢失原始输入；
- 标记分析失败原因和待重试状态；
- 使用默认 30 分钟估时与现有规则生成保守安排；
- 后台在 Codex 恢复后重新分析，但不得擅自改变已经开始或完成的任务。

### 8.2 飞书断线或发送失败

- 业务状态仍写入本地；
- 消息留在发件箱中重试；
- 长连接恢复后继续接收；
- 同一消息或按钮事件重复到达时只处理一次。

### 8.3 Mac 休眠、重启或进程崩溃

- `launchd` 重新启动服务；
- 服务启动时扫描过期提醒和当前计划；
- 已过时的提醒标记为失效；
- 根据当前时间和未完成任务生成恢复计划并发送一张卡片；
- 不连续补发休眠期间的所有提醒。

### 8.4 数据库异常

- SQLite 使用事务和 WAL 模式；
- 写入失败时不向飞书确认成功；
- 每日生成本地备份并保留最近七份；
- 数据恢复后根据事件日志和发件箱继续处理。

## 9. 安全与权限

- 飞书 App Secret、Webhook Secret 和其他凭据只保存在 `.env`，不写入日志、数据库或 Codex 提示词。
- Codex 进程以最小上下文运行，不授予修改任务库的能力。
- 飞书应用只申请接收指定个人聊天或专用群消息、发送消息、处理卡片事件和管理目标任务清单所需的权限。
- 日志默认不记录完整敏感任务内容；错误诊断使用任务 ID 和事件 ID。

## 10. 验证策略

### 10.1 单元测试

- 优先级评分与每日 1–3 个任务上限；
- 时间块容量和冲突处理；
- 两次无响应后的拖延计数、15 分钟动作和重排；
- 卡住、完成、推迟和延期状态流转；
- Codex 返回格式校验与默认降级；
- 休眠恢复时对过期提醒的处理。

### 10.2 集成测试

- 重复飞书消息和重复按钮事件只生成一个任务事件；
- 卡片按钮和文字命令得到相同的状态结果；
- 飞书发送失败进入发件箱并在恢复后成功重试；
- Codex 超时不会阻塞任务保存；
- 重启后能恢复当前任务、提醒和待发消息。

### 10.3 端到端演练

提供可注入的时钟，把完整一天压缩到几分钟，验证：

1. 08:30 计划生成；
2. 任务开始提醒；
3. 两次无响应与最小动作；
4. 午间重排；
5. 完成和卡住反馈；
6. 18:00 收口；
7. 晚间复盘。

最后在当前 Mac 上连续实机运行一天，验证 `launchd` 拉起、网络断开重连和休眠恢复。

## 11. 第一版验收标准

1. 飞书任意自然语言任务能够只入池一次，并自动补全结构化字段。
2. 每天 08:30 自动发送最多 3 个关键任务及明确时间块。
3. 每个时间块开始时发送带按钮的当前任务卡，同时接受文字反馈。
4. 两次各 15 分钟无响应后，系统记录拖延、生成 15 分钟动作并重排当天计划。
5. “完成”“卡住”和“推迟”会立即更新本地状态，并同步飞书展示。
6. 12:00 和 18:00 能依据真实进度自动校准计划。
7. 晚间自动生成当天复盘和明日候选任务。
8. Codex 失败、飞书断线和服务重启均不会丢任务或产生重复业务结果。
9. Mac 唤醒或服务恢复后只发送当前有效的新计划，不轰炸式补发旧提醒。

## 12. 非目标

第一版明确不包含：

- 团队成员任务分配；
- 自动读取日历、CRM、邮件或其他任务系统；
- 云端或多设备部署；
- 周报、月报和长期效率趋势；
- 面向多用户的权限体系；
- 让 Codex 直接写数据库或自主执行工作任务。

## 13. 实施边界

实施应尽量复用现有飞书 WebSocket、任务创建、自然语言入口和优先级测试，但把当前 Markdown 主存储迁移为 SQLite，并保留 Markdown 单向导出。模块之间通过明确的数据结构通信，调度逻辑不得散落在飞书消息处理器中。

本设计批准后，下一步是编写分阶段实施计划；在计划获批前不修改生产功能。
